Coding Actions

Every Command, Context, Window Context and Application Action Interface have a code box where you write the code that will be executed when these actions are activated.

The language used here is MS JScript which is a subset of JavaScript which is, notwithstanding it's limitations, very easy to learn. People with JavaScript experience should have little problem writing code.

The language does not have all the functions you would find while coding JavaScript in a browser, given it is mainly a Plain Jane core JavaScript engine without any libraries aside the core language features.

The language does have many AutoHotKey functions implemented, though not all of them have been tested.

Pressing F1 or Hitting the ? Icon in the Action Dialog with a valid function selected in the Code Box will bring Context Help on the function selected.

Code Box Tools

The tool ribbon under the code box allows you to test and edit the code.

Use the button to execute the code. This is helpful but not always usable depending on the code you are writing. The console window will pop up to handle any send commands. For Testing purpose, you have to manually run the code of all related actions up the chain first If your code box calls a function or uses variables defined in the Application Interface for example, make sure to run that code first.

Use the button to edit the code in an external editor. An external editor such as MS Visual Code or Notepad++ is recommended. The system should automatically update any changes made by the external editor when you save the file. MS Visual Code has an Auto-save Feature that makes this process seamless.

Use the button to set the external editor to the program you want to use.

Use the button to reload the code from the temporary file the external editor is editing if the automatic update is not working.

How It Works

Applications in the tree-view have a set of rules, actions, that define Hot Keys, Speech Rules and code to execute when the action is activated. Each application has it's own JScript Interpreter associated with it.

When an application listed in the tree-view is activated, the code for it is executed. Its code box can be used to define global variables and global functions used by any other actions under that application.

When an action is activated, its code box code is loaded in the Active Script for the Application and the code is executed.

All you need to do is create a function that will encapsulate the code you wish to execute and call that function.

Ultimately you could do without a function and simply have the code in line, but having a function offers the ability to scope the variables your code requires and limit conflicts and have a safer more stable program.

Testing

You can test your Hot Keys and Voice Recognition by CTRL-LEFT CLICKing the Application Item in the tree-view. Make sure the Application Item is not already selected when you do so.

In this mode the Hot Keys and Voice Recognition will activate the tree-view items as it would in the game, however the action code will not run. You have to manually run the code with the code tool ribbon mentioned above.

A good tip is to use Notepad as a test program to test things live, simply change the Application root item to use notepad.

Occasionally the JScript interpreter will fail silently if you call an non existing function. TIP: Select the function in the code editor and Press F1, if the function exists it will pop up in the help. Make sure you respected case sensitivity.

Hot Key Speak (JScript) vs AutoHotKey (AHK)

There are differences between the JScript Code and AutoHotKey Code. Many will go unnoticed but a few have to be mentioned.

Watch for the little things like for example the assign (AKH :=, JScript =) operator and others, not going to go through every minor details here...

Watch for the "\" character when used in a string requiring to be doubled in JScript "\\"

Watch for CASE SENSITIVITY. AHK is not case sensitive while JScript is. Remember this, this is only applicable with function names. Arguments are still case insensitive.

Obviously AHK features/methods had to be ported to work under JScript, be advised that some features were not ported, such as functions requiring a callback...

Some AutoHotKey commands use keywords such as ON, OFF, and so on and some take defined constants like for keys, like ENTER. You may quote these, like "ON", "ENTER", "{ENTER}" and they will resolve properly.

AHK Functions in JScript

Functions in AHK that return a value will be almost identical looking but beware that you MUST call parseInt() or any of the parseFunction() if the return value of the AHK function is not a string. This has to do with the interface between JScript and the AHK original as, mainly, every number in the AHK language is basically a string, hence the reason for the dot (.) operator for concatenating vs the + for adding a value... whereas JScript use + for both features... Note that, if you are just passing values back and forth between AHK functions, you don't need to use parseInt()


;AutoHotKey
Number := Ord(String)
if(Number == 32)
{
    ;space found
    String .= "space"
}
	

//JScript
var Number = parseInt(Ord(String));
if(Number == 32)
{
    //space found
    String += "space";
}

AHK Commands in JScript

As many are aware, a lot of features in AHK come in the form of a command and in the form of a function. None of the AHK features come in command form in JScript. And those commands that did not already come in function form have been wrapped in a function, with pain and misery I might add.

Also, as mentioned, AHK does not distinguish between numbers and strings, so all number arguments may be passed as a string or a number. It's not really important but worth mentioning.


;AutoHotKey
Send, {ENTER}
Sleep, 100
Send, {LBUTTON}
	

//JScript
Send("{ENTER}");
Sleep(100); //Sleep("100"); //would be the same...
Send("{LBUTTON}");

Multiple Output Arguments

Some commands in AutoHotKey are implemented to used multiple output arguments, which allow the command to return multiple pieces of information when executed.

There is a MAJOR difference here between these commands and their JScript implementation you MUST be aware of.

Any command that sets arguments as output arguments had to be re-thought in order to provide the same functionality; without going into much details about the port, faced with the amount of functions to port and the number of issues I faced trying to support output arguments, I decided that these functions and commands would return an associative array.

The associative array returned will hold all the values the command can output, keyed by name where the names match that as specified by the AutoHotKey help file. The associative array names are not case sensitive.

There are limitations to this; for functions that accept prefix tags such as hwndTheHandle which define the output variable TheHandle on the fly, this feature could not be ported and for most cases the functions involved are already superseded by the underlying engine.

When calling such function, you only supply the input arguments and skip over any output arguments.

This example here will make things a little more clear.


;AutoHotKey
filename := "c:\afolder\asubfolder\afile.ext"
SplitPath, %filename%, OutFileName, OutDir, OutExtension, OutNameNoExt, OutDrive
OutputDebug, %OutFileName%
OutputDebug, %OutDir%
OutputDebug, %OutExtension%
OutputDebug, %OutNameNoExt%
OutputDebug, %OutDrive%
	

//JScript
var filename = "c:\\afolder\\asubfolder\\afile.ext";
var ret = SplitPath(filename);
OutputDebug(ret["OutFileName"]);
OutputDebug(ret["OutDir"]);
OutputDebug(ret["OutExtension"]);
OutputDebug(ret["OutNameNoExt"]);
OutputDebug(ret["OutDrive"]);

Single Output Argument

Some commands in AutoHotKey are implemented to use an argument as output parameter, which allow the command to return information through the passed variables name in an argument/parameter.

The solution to this was made much easier than the above Multiple Argument conundrum, the output argument is simply returned by the function, as a regular function would. No associative array required.

When calling such function, you only supply the input arguments and skip over the output argument.

This example here will make things a little more clear.


;AutoHotKey
Random, OutputVar, 0, 100
OutputDebug %OutputVar%
	

//JScript
var ret = Random(0,100);
OutputDebug(ret);

AHK Built-In Variables JScript

Built-In variables support is done via a generic wrapper function that takes the name of the variable as parameter, discussed in more details later


;AutoHotKey
if(ErrorLevel == 5)
{
    OutputDebug %ErrorLevel%
}
	

//JScript

Redesigned Functions

While investigating the functions I wanted to allow calling from JScripts, a few things were found that warranted some tweaks. Some necessary for proper support, some for safer usage.

The following are the AutoHotKey functions that were modified.

BlockInput(Str ONorOFF)

Please Read the BlockInput Help on the AutoHotKey site Here.

This function is supplemented with a timer that will re-enable the inputs after 10 seconds.

If the application is not running with administrator privilege it will wait for the user to release any key he may be holding down. See KeyWaitAll() Below.

GetBuiltInVar

GetBuiltInVar(Str VarName)

Since you cannot access the built-in variables directly, this function will get the value of the specified AutoHotKey variable.

Please Read the Built-In Variable Section on the AutoHotKey site Here.

Note: The function is a blind write, not all variables may be available or useful on the JScript side.

KeyWaitAll

KeyWaitAll()

This function will wait for the user to release all keys he's holding down.

OutputDebug

OutputDebug(Str Text)

Please Read the OuputDebug Help on the AutoHotKey site Here.

This function is set to use the HotKeySpeak Console Window. It can be used as a shorter form of HKS.Debug.Write().

SetTimer

SetTimer(Str FunctionName, Int Time)
SetTimer(Str FunctionName, Str ONorOFForDELETE)

Please Read the SetTimer Help on the AutoHotKey site Here.

The Timer System is slightly different in usage than its AutoHotKey Equivalent.

All timers created are in fact governed by an single internal timer that runs at 33 ms (the fastest I was able to make it tick) so a timer may not be exactly on time.

Instead of passing a callback function the JScript version expects a STRING!!

This String has to be a JScript expression or a function including braces and argument.

This String will be interpreted when the timer event triggers.

This String may reference a GLOBAL variable previously defined by the code, the variable value will be resolved when the timer event triggers. If you want the value to be interpreted when you setup the timer, you need to build the string yourself.

This String is used internally as a key to map to the internal structure each timer instance requires, It is used to locate the timer if you want to turn it OFF. It is also used by the other functions described later.

Conventional example, AHK vs JScript


;AutoHotKey
SetTimer, TimeFunc, -100 ;-time means trigger only once
TimeFunc()
{
    OutputDebug, TimeFunc Triggered
}
global tickcount := 0
SetTimer, TimerLabel, 100
return

TimerLabel:
{
    OutputDebug, TimerLabel Triggered 
    tickcount++
    if(tickcount > 5)
    {
        SetTimer, TimerLabel, OFF
    }
}
	

//JScript
SetTimer("TimeFunc()", -100); //-time means trigger only once
function TimeFunc()
{
    OutputDebug("TimeFunc Triggered");
}
var tickcount = 0;
SetTimer("TimerLabel()", 100);

function TimerLabel()
{
    OutputDebug("TimerLabel Triggered");
    tickcount++;
    if(tickcount > 5)
    {
        SetTimer("TimerLabel()", "OFF");
    }
}

SetTimer accepts functions with argument and system functions


//JScript
function DoIt(Key)
{
    Send(Key);
}
SetTimer("DoIt(1)", -100); //since 1 (a number) will resolve as "1" internally, this is acceptable
SetTimer("Send(2)", -200);
SetTimer("Send(3)", -300);
SetTimer("DoIt('4')", -400); //same result
SetTimer("Send('H')", -500); //Obviously if wanting to send a character...
SetTimer("Send('ello')", -500); //... or a string, the double quote and single quote trick works
SetTimer("Send('{ENTER}')", -600);

SetTimer accepts global variables

Example stacking events that have a cool down system using a timer. For systems that prevent keyboard bashing for example, like you can't fire magic 2 if you just fired magic 1. The cool down here is not the amount of time Magic 1 takes to recharge.


//JScript
//In Application Code
var CoolDownTime = 1000;
var TimersAdded = [];
//Callback to keep time, runs when there is no more cool down accumulated
function CoolDownOver()
{
    try //Try to do this...
    {

        //The timer finally had a chance to trigger
        //loop through all the registered stacked timers and delete them
        while(TimersAdded.length)
        {
            //get the timer name and remove from array
            var timername = TimersAdded.pop();
            //delete the timer
            SetTimer(timername, "Delete"); 
        }
        //Or if you don't have other timers in the rules, this would do
        //TimersAdded = []; // or reuse current... while(TimersAdded.length) {TimersAdded.pop()}
        //TimerClearAll();
    }
    catch(e) //...catch any problems
    {
        
    }
}
//pass "Send('AKey')" for example
function StackSendAction(Code) 
{
    try //Try to do this...
    {
        //get the time left in the cool down
        var TimeLeft = parseInt(TimerTimeLeft("CoolDownOver()"));
        //no cool down period awaiting...
        if(TimeLeft == 0)
        {
            //execute the code passed NOW via eval
            eval(Code); 
            //set the cool down
            SetTimer("CoolDownOver()", -CoolDownTime); 
        }
        else
        {
            //there is a cool down in progress
            //create a "Send(key)//1..2..3.." unique timer to run this code later as a timer
            //the comment appended to it will serve as a unique key. which will allow stacking the same code multiple times
            var TimerCodeName = Code + "//" + TimersAdded.length;
            //remember the timer created
            TimersAdded.push(TimerCodeName); 
            //create the timer
            SetTimer(TimerCodeName, -TimeLeft); 
            //add to the cool down timer
            SetTimer("CoolDownOver()", -(TimeLeft + CoolDownTime)); 
        }
    }
    catch(e) //...catch any problems
    {
        
    }
}
//In An Action
StackSendAction("Send('a')");

//In Another Action
StackSendAction("Send('b')");

Helper Classes

Aside the Redesigned Functions mentioned above, HotKeySpeak also extents AutoHotKey with a set of classes that give access to HotKeySpeak inherent features as well as providing features you may find useful.

HKS Root Class

The HKS class is the root container class for all HotKeySpeak features.

From the HKS class you can access sub classes categorized by feature.

This class has no member functions.

HKS.Action Class

The Action class provides access the actions, which are the Commands, the Command Context and other items found in the Application TreeView.

HKS.Action.Activate Member Function

HKS.Action.Activate(Str ActionName)

The Activate function takes an Action Name as defined in the Application Tree-view.

When called, the action will be activated as though its hot key was pressed or spoken sentence was uttered.

The actions allowed are limited to Commands and Command Context.

The search is performed from the root application Item.

If the action is or is under a Window Context the action will not activate.

Disabled actions or ones in disabled groups cannot be activated.

Actions contained in a hierarchy of Command Contexts will not have the parent Command Contexts activated. You will need to activate each parent action individually if required.


//JScript
//Activate the FileMenu Context
HKS.Action.Activate("FileMenu");

HKS.Action.Run Member Function

HKS.Action.Run(Str ActionName)

The Run function takes an Action Name as defined in the Application Tree-view.

This method allows running code of an action directly which is sometimes useful.

When called, the code in the specified action will be executed without activating the action.

Any tree-view item, aside the root application item, that has code can be executed this way.

The search is performed from the root application Item.

This method ignores the Enabled flag of the Action or the Enabled flag of any parent group.

Actions contained in a hierarchy of Command Contexts will not have the parent Command Contexts code executed. You will need to run each parent action code individually if required.


//JScript
//Run the FileSave Command
HKS.Action.Run("FileSave");

It was mentioned before that you can define functions in the Application Action and call them directly from many places; you may have a set of functions or features located in a library you would like to use, without wanting to copy and paste the code, creating the functions, in the Application Action. HKS.Mouse Class

The Mouse Class offers functions and sub classes that will allow you to control the mouse.

These supplemental classes were added to provide extra support found to be essential especially when trying to add mouse movement for games.

HKS.Mouse.StartDetect Member Function

HKS.Mouse.StartDetect()

Turn the mouse movement detection back ON so the game sees the mouse moving.

Requires you having called HKS.Mouse.StopDetect() first.


//JScript
//Stop the app from seeing mouse movement
HKS.Mouse.StopDetect();
//Invisibly Move The Mouse here
...
//Allow the app to see mouse movement again
HKS.Mouse.StartDetect();
//Move The Mouse here
...

HKS.Mouse.StopDetect Member Function

HKS.Mouse.StopDetect()

Turn the mouse movement detection OFF so the game can't see the mouse moving.

Requires you call HKS.Mouse.StartDetect() when you are done otherwise the user may not be able to click the app again without first clicking another app on the desktop.

NOTE: Movement made with HKS.Mouse.Direct functions may still be detected by the game. Use Mouse.Pix or Mouse.UV to move the mouse undetected.


//JScript
//Stop the app from seeing mouse movement
HKS.Mouse.StopDetect();
//Invisibly Move The Mouse here
...
//Allow the app to see mouse movement again
HKS.Mouse.StartDetect();
//Move The Mouse here
...

HKS.Mouse.Direct Class

The Mouse.Direct Class offers functions to move the mouse directly via the hardware.

Some games read the mouse movement directly from the hardware as opposed to determine a movement off of the mouse moving on the game window.

For these games you have to send mouse movement through hardware for them to notice.

These functions also move the mouse over the window, meaning they are compatible for regular windowed application though it is recommended you use the Mouse.Pix or mouse.UV for those.

The Direct functions take Mickeys for movement distance, not Pixels. A mickey is the amount that a mouse has to move for the mouse to report that it has moved.

HKS.Mouse.Direct.MoveByXYVector Member Function

HKS.Mouse.Direct.MoveByXYVector(Int MickeysDX, Int MickeysDY, Int NumMoves)

Moves the mouse in the direction specified by Deviation X, Deviation Y vector. Cannot be a fraction.

NumMoves specifies the number of times the deviation is applies. Must be 1 or higher.

The actions takes 16 ms per repeat to perform.

The movement is relative to the current mouse pointer position, if used in the context where an actual mouse pointer is used.


//JScript
//move 5000 Mickeys Upper Left... Should be enough to set it upper left corner

Sleep(1000);
//move 200 Mickeys to the right moving by 20, 10 times
HKS.Mouse.Direct.MoveByXYVector(20, 0, 10);
Sleep(1000);
//move 200 Mickeys down, a little math helps clarify
HKS.Mouse.Direct.MoveByXYVector(0, 200/10, 10);
Sleep(1000);
//move 200 Mickeys left
HKS.Mouse.Direct.MoveByXYVector(-200/10, 0, 10);
Sleep(1000);
//move 200 Mickeys up
HKS.Mouse.Direct.MoveByXYVector(0, -200/10, 10);
Sleep(1000);

HKS.Mouse.UV Class

The Mouse.UV Class offers functions to move the mouse by UV value.

UV is a coordinate system use in graphical systems. UV defines a position in percentage form via a 0 through 1 value. 0,0 would be upper left. .5,.5 would be center. 1,1 would be bottom right.

Using UV you can be accurate in positioning the mouse on a window regardless of it's size. Or in a game regardless of the resolution it's running at. And so your code can work no matter the resolution chosen.

Iterations is the number of iterations it will take to reach the destination, this is similar to the Speed concept in AutoHotKey's mouse move function.

The actions takes 16 ms per iteration to perform.

NOTE: The distance to move converted to pixel divided by the iteration must resolve in a pixel value that is not a fraction else the mouse may stay in place because it can't move by a distance of 0.1 for example.

The Recognition Class allows you to get information related to the status of the Voice Recognition, such as sentences detected, commands detected and so on.

Returns the dictation text as detected by the voice recognition system while in Dictation Mode.

See Dictation example for use.

Returns an array of all the Commands that have been detected on the voice recognition process currently in progress.

HKS.Recognition.GetCommandString()

All Commands are separated with the pipe | character .

Returns the average Confidence Level of the voice recognition process currently in progress.

HKS.Recognition.GetConfidencesArray()

Returns an array of Confidences for all the Commands that have been detected on the voice recognition process currently in progress.

See NOTE: in example for access.

See GetConfidenceThreshold.

See Confidence Level Override Example.

 
HKS.Recognition.GetConfidenceThreshold()

Returns the Confidence Threshold as set in the Options so you can use it to refine you coded actions based on confidence.

When the average confidence level falls under this value, actions are not execute.

When the average confidence is above this value, actions are executed...

There are occasion where a specific command has a lower confidence than the average, but not enough so to cause the average to drop below the threshold and so the command may execute... The balancing act between word restrictions, fast and loose on the fly processing and noise may cause the system to think a noise is a valid command, but with a confidence not low enough to cause the average to drop. This is especially true if you defined one syllable words. You may override the system for such words and prevent misfiring.

That sound silly! You may ask yourself why the system uses an average confidence level as opposed to the actual word confidence available. The reason was determined during trial that the confidence drops when issuing a number of commands in a row, enough for the commands to be lower in confidence while the overall confidence is acceptable.

See NOTE: in example for access.

See Confidence Level Override Example.

 
HKS.Recognition.GetSentencesArray()

Returns an array of all the sentences that have been detected on the voice recognition process currently in progress.

See NOTE: in example for access.

 
HKS.Recognition.IsDictating()

Returns true if the voice recognition system is in Dictation Mode, false if Not.

See Dictation example for use.

 
HKS.Recognition.StartDictating(Str "CallbackFunction()")

Turns the Dictation Mode ON.

Dictation Mode turns OFF the Command System and will now recognize every word spoken by the user. This is used to allow speaking sentences that the user would normally type, like in a chat box for example.

The function takes a callback function but in a string fashion like SetTimer.

The callback function will be called for every word detected. Since the system is asynchronous this function is where you will do all the work to send the text to the game or application.

The callback function will also be called when Dictation Mode ends. You can use IsDictating() to see the context of the call.

Dictation Mode will turn off automatically when the user stops talking after having dictated a sentence, so basically is it ON for one sentence.

Note: If a request to turn on Dictation was made in the middle of a long series of commands, Dictation Mode will not turn ON right away, all existing commands in the queue be ignored and Dictation Mode will start when the user stops and restarts talking.

Note: The Hot Key system will be disabled while Dictation Mode is ON.

See Dictation example for use.

//JScript
//In Application Action Code
function DebugCommands()
{
    OutputDebug("****************");
    //CONFIDENCE
    OutputDebug("Confidence: " + HKS.Recognition.GetConfidence());
    OutputDebug("----------------");

    //COMMANDS ARRAY
    var arr = HKS.Recognition.GetCommandsArray();
    var confarr = HKS.Recognition.GetConfidencesArray();
    //NOTE: use .Length() for AHK array as opposed to .length for JS arrays
    var NumCommands = arr.Length(); 
    OutputDebug("Sequence Num Commands: " + NumCommands)
    //NOTE: AHK array goes from 1 to Length() Inclusive, as opposed to javascript's 0 to length-1
    var FinalCommandInSeries = arr[NumCommands];
    var FinalCommandCondidence = confarr[NumCommands];
    OutputDebug("Sequence Last Command: " + FinalCommandInSeries);
    OutputDebug("Sequence Last Command Confidence: " + FinalCommandCondidence);
    
    OutputDebug("Sequence All Commands: ");
    for(var i = 1; i<=NumCommands; i++)
    {
        OutputDebug(arr[i] + " Conf: " + confarr[i]);
    }
    OutputDebug("----------------");

    //SENTENCES ARRAY
    var arr = HKS.Recognition.GetSentencesArray();
    //NOTE: use .Length() for AHK array as opposed to .length for JS arrays
    var NumSentences = arr.Length(); 
    OutputDebug("Sequence Num Sentences: " + NumSentences)
    //NOTE: AHK array goes from 1 to Length() Inclusive, as opposed to javascript's 0 to length-1
    var FinalSentenceInSeries = arr[NumSentences];
    OutputDebug("Sequence Last Sentence: " + FinalSentenceInSeries);
    OutputDebug("Sequence All Sentences: ");
    for(var i = 1; i<=NumSentences; i++)
    {
        OutputDebug(arr[i])
    }
    OutputDebug("----------------");
}

Confidence Level Override Example.


//JScript
//send 1 if "One" is spoken, "One" being a "not recommended" one syllable word, but this is a way around this 
//Breathing in the microphone with the nose often makes the system think "One" has been said
//This issue usually happens in the context of a single issued command
//This code will check is the context is a single command and double check if the confidence level for the command specifically passes the threshold on its own

//get the confidence array
var confarr = HKS.Recognition.GetConfidencesArray();
//the number of commands derived from the array length
var NumCommands = confarr.Length();
//is this a single command?
if(NumCommands == 1)
{
    //only allow high confidence to take action, a true spoken "One"
    var MyConfidence = confarr[NumCommands];
    if(MyConfidence >= HKS.Recognition.GetConfidenceThreshold())
    {
        Send(1);
    }
}
else
{
    //detected as part of a series of commands, it's safe to run
    Send(1);
}

//JScript
//the extracted sound resources
var ActivateSound = ""; 
var ReloadSound = ""; 
var SwitchGunSound = ""; 
//Define a Unique Function Name
function OnSetup()
{
    //To safely do things without an error message poping up
    try //Try to do this...
    {
        //Extract the Sound resources
        ActivateSound = HKS.Resource.Fetch("confirm.wav");
        ReloadSound = HKS.Resource.Fetch("reload.wav");
        SwitchGunSound = HKS.Resource.Fetch("switch.wav");
        //Play the sound when application gets focused
        PlaySound(ActivateSound);
    }
    catch(e) //...catch any problems
    {
        //debug out at Level 3, Ignorable Error Level
        HKS.Debug.WriteNL("", 3);
        HKS.Debug.WriteNL("Exception:", 3);
        HKS.Debug.WriteNL("Game OnSetup()", 3);
        HKS.Debug.WriteNL("" + e, 3);
    }
}
//Call Function
OnSetup();

//This function is called to play sound by various actions that need a clue that they triggered
function PlayTheSound(sound)
{
    try
    {
        SoundPlay(sound ,"");
    }
    catch(e)
    {
    }
}

//In Reload Action
PlayTheSound(ReloadSound);

//In SwitchGun Action
PlayTheSound(SwitchGunSound);

You can find the MS Voice Agent Text To Speech XML specifications HERE

 
HKS.VoiceAgent.Stop()

Interrupts the current sentence being spoken by MS Voice Agent Text To Speech.

Example showing the result for every Member Function.


//JScript
HKS.VoiceAgent.Say("Hello, I am the MS SAPI text To Speak System");
OutputDebug("I'm Saying Hello");
HKS.VoiceAgent.Say("How are you doing this fine day??");
OutputDebug("I'll say How are you Next...");
Sleep(8000);
OutputDebug("Now I'll say I am Fine and You? and wait");
HKS.VoiceAgent.SayWait("I am Fine and You?");
OutputDebug("I just Said I am Fine and You...");
Sleep(2000);

HKS.WinAPI Class

The WinAPI Class has WinAPI Functions defined that may be useful writing rules.

NOTE: Some functions don't have the same number of arguments as the original Microsoft documentation, still the information from the original source will be linked as reference.

NOTE: Some of the WinAPI functions that have a AutoHotKey Equivalent may not be included.

NOTE: Functions that only apply to the calling thread are not included as they will only affect the HotKeySpeak UI.

NOTE: Functions that were judged not useful or too complex are not included.

The WinAPIs here are split by the same categories as Microsoft Reference Pages which sometimes has functions in odd places.
